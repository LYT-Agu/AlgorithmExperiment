# 排序实验结果分析

---

## 一、实验任务①：等价类理解（n=100，两组随机数据）

### 1.1 实验设计

本实验生成两组各含 100 个整数的随机数序列（取值范围 [0, 9999]），分别使用 seed=42 和 seed=777 作为随机种子。对每组数据分别执行冒泡排序、归并排序和快速排序三种算法，记录每种算法执行过程中的**比较操作总次数**，以此观察同一规模、不同数据对算法行为的影响。

### 1.2 实验数据

**表1：两种排序在两组随机数据上的比较次数对比**

| 数据组 | 随机种子 | 冒泡排序 | 归并排序 | 快速排序 |
|--------|----------|----------|----------|----------|
| Group 1 | 42 | **4845** | **540** | **892** |
| Group 2 | 777 | **4830** | **545** | **906** |

### 1.3 结果分析

**(1) 冒泡排序：比较次数接近理论最坏值，提前终止效果有限**

冒泡排序在两组数据上的比较次数分别为 4845 次和 4830 次，两者仅相差 15 次（相对偏差约 0.31%）。对于 n=100 的输入，冒泡排序的理论最大比较次数为 C(100, 2) = 4950 次。实验值与该理论上限的比值分别为 97.88% 和 97.58%，说明：

- 两组均为**近反序分布**的随机数据，提前终止标志 `flag` 在大部分外层循环中未被触发；
- 冒泡排序的比较次数主要由输入规模决定，受具体数据排列的影响较小——这符合 O(n²) 时间复杂度的特征，即比较次数的增长由嵌套循环的结构决定，而与数据的"好坏程度"关系有限。

**(2) 归并排序：比较次数高度稳定，波动极小**

归并排序在两组数据上的比较次数分别为 540 次和 545 次，相差仅 5 次（相对偏差约 0.93%）。这一现象的原因在于：

- 归并排序采用严格的**均匀二分分解策略**，递归树的结构完全由输入规模 n 决定，与数据的数值内容无关；
- 比较操作仅在 merge 合并阶段发生，而 merge 过程中的比较次数取决于两个子数组中元素的大小关系，但由于每次 merge 的子问题规模固定（由二分结构决定），总的比较次数在不同数据间仅有微小波动；
- 这说明归并排序具有**良好的稳定性**——在同一等价类内，其计算代价可预测。

**(3) 快速排序：比较次数差异显著，pivot 选择是关键变量**

快速排序在两组数据上的比较次数分别为 892 次和 906 次，相差 14 次（相对偏差约 1.57%）。虽然绝对差值不大，但相较于归并排序的 0.93%，快排的波动幅度明显更大。原因如下：

- 本实现选取**首元素作为 pivot**，分区质量完全取决于首元素在当前子数组中的秩（rank）；
- 不同随机数据导致 pivot 在各次递归中的位置不同，从而产生不同的分区均衡度；
- 当 pivot 偏离中位数时，左右子问题规模不均等，导致较深的一侧产生更多递归调用和比较操作。

### 1.4 等价类结论

综合以上三组数据可以得出：

> **同一规模（n=100）、同分布（均匀分布 [0, 9999]）的随机数据构成一个等价类**。在此等价类上：
> - 冒泡排序的行为高度一致（O(n²) 主导，提前终止贡献微弱）；
> - 归并排序的行为几乎不变（分治结构唯一确定）；
> - 快速排序的行为存在可控范围内的波动（依赖 pivot 分区质量）。
>
> 这一实验验证了等价类的核心思想：算法在同一等价类内的性能表现具有统计一致性，可用于预测和评估算法的实际效率。

## 二、实验任务②：不同规模的比较次数对比

### 2.1 实验设计

本实验选取 7 个不同规模的数据集（n = 10、100、1000、2000、5000、10000、100000），每个规模使用相同种子（seed=2026）生成的均匀随机数据，分别运行三种排序算法，记录比较操作次数，以观察随输入规模增长时各算法的时间复杂度变化趋势。

### 2.2 实验数据

**表2：不同输入规模下三种排序算法的比较次数**

| 输入规模 N | 冒泡排序 | 归并排序 | 快速排序 | 冒泡/归并比值 | 快排/归并比值 |
|------------|----------|----------|----------|---------------|---------------|
| 10 | 42 | 22 | 34 | 1.91× | 1.55× |
| 100 | 4,929 | 544 | 914 | 9.06× | 1.68× |
| 1,000 | 499,004 | 8,735 | 14,962 | 57.13× | 1.71× |
| 2,000 | 1,998,054 | 19,472 | 33,492 | 102.61× | 1.72× |
| 5,000 | 12,481,390 | 55,268 | 93,977 | 225.86× | 1.70× |
| 10,000 | 49,958,685 | 120,476 | 200,259 | 414.79× | 1.66× |
| 100,000 | 跳过 | 1,536,220 | 2,860,233 | — | 1.86× |

*注：N=100000 时冒泡排序因时间开销过大被跳过。*

### 2.3 结果分析

**(1) 冒泡排序：严格符合 O(n²) 增长规律**

从表中数据可以看出，冒泡排序的比较次数增长呈现明显的**二次函数特征**：
- N 从 10 → 100（扩大 10 倍）：比较次数从 42 → 4,929（扩大约 **117.6 倍** ≈ 10²）
- N 从 100 → 1,000（扩大 10 倍）：从 4,929 → 499,004（扩大约 **101.2 倍** ≈ 10²）
- N 从 1,000 → 10,000（扩大 10 倍）：从 499,004 → 49,958,685（扩大约 **100.1 倍** ≈ 10²）

每当输入规模扩大 10 倍，比较次数大约扩大 100 倍，精确验证了冒泡排序的 **Θ(n²)** 时间复杂度。此外，N=100 时实际比较次数 4,929 与理论最大值 4,950 的比值为 99.58%，表明对于大规模随机数据，提前终止优化的效果趋近于零——这是因为在 n 足够大时，几乎必然存在大量逆序对，`flag` 标志极少被触发。

当 N 达到 100,000 时，按 Θ(n²) 估计比较次数将高达约 **5×10⁹** 量级，这在常规硬件上已经无法在合理时间内完成，因此程序将其标记为 "skipped"。这一事实直观地展示了 O(n²) 算法在大规模数据处理上的**不可行性**。

**(2) 归并排序：稳定遵循 O(n log n) 增长规律**

归并排序的比较次数增长呈现出典型的 **n log n 特征**：
- N 从 10 → 100：22 → 544（扩大 **24.7 倍**）
- N 从 100 → 1,000：544 → 8,735（扩大 **16.1 倍**）
- N 从 1,000 → 10,000：8,735 → 120,476（扩大 **13.8 倍**）
- N 从 10,000 → 100,000：120,476 → 1,536,220（扩大 **12.8 倍**）

随着规模增大，倍数逐渐收敛至 log₁₀(10) × (log₂N 变化部分)，整体符合 **Θ(n log n)** 复杂度。特别值得注意的是，即使 N=100,000 这样的大规模，归也仅需约 **156 万次**比较即可完成——相比冒泡排序在 N=10,000 时的近 5,000 万次，效率差距已达 **30 倍以上**。若冒泡排序强行运行至 N=100,000，两者的效率差距将达到 **数千倍级别**。

进一步验证 n log n 关系：以 N=10,000 为基准，理论上 T(n) ∝ n·log₂n：
- T(10000)/T(10000) = 1
- T(100000)/T(10000) = 1,536,220 / 120,476 ≈ **12.75**
- 而 (100000·log₂100000)/(10000·log₂10000) = 10 × (16.61/13.29) ≈ **12.50**

理论与实测比值高度吻合，充分证明了归并排序的 **Θ(n log n)** 时间复杂度。

**(3) 快速排序：平均情况为 O(n log n)，但常数因子大于归并排序**

快速排序同样表现出 **n log n 级别** 的增长趋势：
- N=10,000 时比较次数为 200,259 次，约为归并排序（120,476 次）的 **1.66 倍**；
- N=100,000 时比较次数为 2,860,233 次，约为归并排序（1,536,220 次）的 **1.86 倍**；

快排的常数因子始终大于归并排序，主要原因在于：
- **pivot 选择策略**：本实现选取首元素作为 pivot，在随机数据下期望位于中间位置的概论为 50%，但并非最优选择（如三数取中或随机化 pivot 可改善）；
- **partition 过程的比较方式**：快排在 partition 中需要同时进行左端和右端的扫描比较，且每趟 partition 至少需要 (r-l+1) 次比较，而归并的 merge 操作在合并有序子数组时更为高效。

尽管常数因子较大，快速排序的实际运行速度通常仍优于归并排序（因为其空间复杂度为 O(log n) 而非 O(n)，且 cache locality 更好），但在**纯比较次数**这一度量指标上，本次实验中快排确实略逊于归并。

**(4) 三种算法的综合对比与复杂度层次划分**

通过表2中的"冒泡/归并比值"一列可以清晰看到：
- N=10 时比值仅为 1.91×，三种算法差距不明显（小规模下 O(n²) 与 O(n log n) 差异不显著）；
- N=1,000 时比值已达 57×，**复杂度的阶差开始主导**；
- N=10,000 时比值飙升至 415×，O(n²) 与 O(n log n) 已属**完全不同的数量级**；

这一对比有力地说明了：**算法的时间复杂度阶数是决定其在大规模数据上性能的根本因素，低阶算法的任何常数级优化都无法弥补阶数劣势。**

## 三、实验任务③：子问题规模分析与递归结构

### 3.1 实验设计

本实验在归并排序和快速排序的递归过程中，记录每次进入递归函数时所处理的子问题规模（即当前待排序区间长度），用于分析两种分治算法的**递归结构特征**和**子问题规模分布模式**。

### 3.2 实验数据

**表3：归并排序与快速排序的递归调用次数对比**

| 输入规模 N | 归并排序递归次数 | 快速排序递归次数 | 比值（归并/快排） |
|------------|------------------|------------------|-------------------|
| 10 | 9 | 6 | 1.50× |
| 100 | 99 | 67 | 1.48× |
| 1,000 | 999 | 674 | 1.48× |
| 2,000 | 1,999 | 1,347 | 1.48× |
| 5,000 | 4,999 | 3,403 | 1.47× |
| 10,000 | 9,999 | 6,949 | 1.44× |
| 100,000 | 99,999 | 82,771 | 1.21× |

**表4：各规模下的前若干子问题规模（节选）**

| 规模 | 算法 | 前20个子问题规模 |
|------|------|-------------------|
| N=10 | 归并 | 10 → 5 → 3 → 2 → 2 → 5 → 3 → 2 → 2 |
| N=10 | 快排 | 10 → 6 → 2 → 3 → 2 |
| N=100 | 归并 | 100 → 50 → 25 → 13 → 7 → 4 → 2 → 2 → 3 → 2 → ... |
| N=100 | 快排 | 100 → 76 → 8 → 6 → 3 → 2 → 2 → 67 → 64 → 6 → ... |
| N=1000 | 归并 | 1000 → 500 → 250 → 125 → 63 → 32 → 16 → 8 → 4 → 2 → ... |
| N=1000 | 快排 | 1000 → 723 → 236 → 106 → 82 → 77 → 66 → 39 → 9 → 8 → ... |
| N=10000 | 归并 | 10000 → 5000 → 2500 → 1250 → 625 → 313 → 157 → 79 → 40 → 20 → ... |
| N=10000 | 快排 | 10000 → 6995 → 3751 → 2454 → 1637 → 1484 → 1056 → 930 → ... |
| N=100000 | 归并 | 100000 → 50000 → 25000 → 12500 → 6250 → 3125 → 1563 → 782 → ... |
| N=100000 | 快排 | 100000 → 69269 → 18041 → 16210 → 3252 → 2319 → 1975 → ... |

### 3.3 结果分析

**(1) 归并排序：严格均匀二分，递归结构完全确定性**

从表3可以看出，归并排序的递归调用次数满足以下精确公式：

$$\text{递归调用次数} = 2 \times \lceil N/2 \rceil - 1 \approx N - 1$$

例如：
- N=100 时递归次数恰好为 99（= 100 − 1）
- N=10,000 时递归次数恰好为 9,999（= 10,000 − 1）
- N=100,000 时递归次数恰好为 99,999（= 100,000 − 1）

这是因为归并排序采用**严格的二分分解策略**：每次将当前区间 `[l, r]` 以中点 `mid = ⌊(l+r)/2⌋` 划分为 `[l, mid]` 和 `[mid+1, r]` 两半，直到子问题规模为 1 时终止递归。这种分解产生的递归树是一棵**近似完全二叉树**，总结点数为 2⌈log₂N⌉−1 个内部节点加上叶子节点，总递归调用次数恒等于 **N−1**（不含规模为 1 的基情形）。

从表4的子问题规模序列可以观察到归并排序的分解模式：
- **第一层**：规模为 N（如 100000）
- **第二层**：规模约为 N/2 和 N/2（如 50000、50000）
- **后续层**：依次为 N/4、N/4、N/4、N/4……呈严格的**2 的幂次递减**
- **底层**：大量规模为 2 或 3 的小子问题

这种均匀分解保证了每一层的总工作量均为 **Θ(n)**，共有 **Θ(log n)** 层，故总时间复杂度为 **Θ(n log n)**。同时由于分解过程与数据内容无关，归并排序的递归结构是完全确定的——这也是其在任务①中表现出极小波动性的根本原因。

**(2) 快速排序：分区不均衡导致递归结构不确定**

与归并排序形成鲜明对比的是，快速排序的递归调用次数**远小于 N−1**，且没有简单的闭式公式：

- N=100 时快排仅递归 67 次（归并为 99 次，减少 32.3%）
- N=1,000 时快排递归 674 次（归并为 999 次，减少 32.5%）
- N=100,000 时快排递归 82,771 次（归并为 99,999 次，减少 17.2%）

减少的原因在于：快速排序在 partition 之后产生的**两个子问题的规模之和等于原问题规模减 1**（pivot 元素已就位，不再参与后续递归），而非归并排序的两个子问题规模之和等于原问题规模。这意味着快排每层处理的元素总数逐层递减，递归树的节点数自然少于归并排序。

更关键的特征体现在**子问题规模的不均匀性**上（见表4）：

以 N=100 的快排为例，前几个子问题规模为：
```
100 → 76 → 8 → 6 → 3 → 2 → 2 → 67 → 64 → 6 → ...
```

可以明显观察到：
- 第一次 partition 将规模 100 分解为 **76** 和 **23**（≈ 100−1−pivot），严重右偏；
- 第二次处理规模 76 的子问题时，又分解出 **67** 和 **8**，依然很不均衡；
- 这种**长短交替出现**的模式贯穿整个递归过程，与归并排序整齐的"折半"形成强烈对比。

以 N=100,000 为例，第一个 pivot 将问题分为 **69,269** 和 **30,730**（比例约 2.26 : 1），远非理想的 1:1 均匀分割。这种不均衡导致了：
- **较深的递归深度**：最长路径上的子问题规模衰减缓慢；
- **较大的常数因子**：不均衡意味着某些层的工作量偏大（尽管总量仍为 O(n)）；
- **数据敏感性**：不同数据会导致不同的分区序列，这就是任务①中快排波动最大的原因。

**(3) 递归深度与空间复杂度的隐含信息**

虽然本实验未直接测量递归栈深度，但可以从子问题规模序列推断：

- **归并排序**的最大递归深度为 **⌈log₂N⌉**：
  - N=100,000 时深度约为 17 层（2¹⁶=65536 < 100000 < 131072=2¹⁷）
  
- **快速排序**的最大递归深度取决于最坏分区情况：
  - 从 N=100,000 的子问题序列看，左侧分支经历了 100000→69269→18041→... 的衰减，深度明显高于归并排序
  - 若每次都选到极端 pivot（最大或最小元素），最坏情况下递归深度可达 **O(n)**，这将导致栈溢出风险

这也解释了为什么工程实践中常用**尾递归优化**或**限制递归深度后切换插入排序**等技术来增强快排的鲁棒性。

### 3.4 综合结论

通过对子问题规模的系统记录和分析，本任务揭示了两种分治排序算法的本质区别：

| 对比维度 | 归并排序 | 快速排序 |
|----------|----------|----------|
| 分解策略 | 均匀二分（中点划分） | 按 pivot 分区（位置不确定） |
| 递归结构 | 完全确定性（仅取决于 N） | 数据依赖型（取决于 pivot 选择） |
| 递归调用次数 | 恒为 N−1 | 小于 N−1，无闭式公式 |
| 子问题规模分布 | 严格 2 的幂次递减 | 不规则分布，可能严重失衡 |
| 最坏递归深度 | ⌈log₂N⌉（稳定） | O(n)（理论上界，概率极小） |
| 空间复杂度 | O(n)（辅助数组） | O(log n)（平均递归栈） |

## 四、总体结论

本实验通过三个递进式的任务，从微观（等价类）、宏观（规模增长）和内部结构（子问题分解）三个维度全面考察了冒泡排序、归并排序和快速排序三种经典排序算法的性能特征。主要结论如下：

1. **时间复杂度的阶数决定了算法的可扩展性**：O(n²) 的冒泡排序在小规模（n≤100）下尚可接受，但当 n > 10,000 后已完全不实用；O(n log n) 的归并排序和快速排序则能轻松应对百万级数据。

2. **同阶算法间的常数因子差异不容忽视**：归并排序和快速排序同为 O(n log n)，但归并排序的比较次数 consistently 少于快速排序（约少 40%~50%），这说明 pivot 选择策略和 partition 实现细节对实际性能有显著影响。

3. **等价类概念提供了算法分析的统一视角**：同一规模、同分布的数据构成一个等价类，算法在该等价类上的行为具有统计一致性——归并排序的一致性最强，快排次之，冒泡排序居中。

4. **分治策略的选择深刻影响了算法特性**：归并排序的"先分后合"保证了均匀性和稳定性，但需要额外空间；快速排序的"先分再治"实现了原地排序，但引入了数据依赖性和潜在的最坏情况风险。二者各有适用场景，需根据实际问题权衡选择。
